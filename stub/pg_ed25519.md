`pg_ed25519` provides `ed25519.sign` and `ed25519.verify` overloads for `bytea` and `text` messages. Signing accepts a 32-byte public key and a 32-byte private seed and returns a 64-byte signature; verification needs the message, signature, and public key.

### Sign and Verify

```sql
CREATE EXTENSION pg_ed25519;

WITH keys AS (
  SELECT
    decode('NZV4l8hck3iUqInENyI+nn5vkW7rqzQg0uiuuZkPnHE=', 'base64') AS public_key,
    decode('eT7qaT8vkIgCl6/9EmEDYYEgxA0oOgHc0P6UYzcQN28=', 'base64') AS private_key
), signed AS (
  SELECT public_key,
         ed25519.sign(text 'some data for signing', public_key, private_key) AS signature
  FROM keys
)
SELECT ed25519.verify(text 'some data for signing', signature, public_key)
FROM signed;
```

The example key pair is public test material from upstream and must never be reused.

### Caveats

- The extension is not a key generator or keystore. Passing private material through SQL can expose it in application logs, statement logging, monitoring, or `pg_stat_activity`; use controlled signing infrastructure for real secrets.
- Version `0.2` bundles its own Ed25519 and SHA-512 C implementation. Independently test signatures against a maintained cryptographic library before interoperability or security use.
- Upstream claims PostgreSQL 9.4 and later but has no current compatibility or security-maintenance statement. Treat immutable function declarations as planner metadata, not an assurance about secret handling.
